## 1. draft.ts — обязательное начало с заголовка и призыв в конце

### Реализация

В `userContent` после блока с фактами добавлена строка:

```
Обязательно: начни статью с заголовка (указан выше). В конце статьи добавь призыв перейти на сайт и оформить заказ.
```

Далее без изменений идёт инструкция по объёму и подзаголовкам. Промпт 2 по-прежнему в `system` с подстановкой `{role}`; при пустом Промпте 2 используется дефолтный system.

### Проверки

- **Пустой headline:** в user передаётся `Заголовок: ""`. Модель получает явное указание «начни с заголовка (указан выше)» — при пустом заголовке результат может быть формальным; логика не падает.
- **Пустой facts:** блок «Проверенные факты» может быть пустым; инструкция «опираясь на эти факты» остаётся. Приёмлемо.
- **Промпт 2 пустой:** `systemContent` берётся из дефолта; обязательная структура задаётся в user — дублирования нет, конфликта нет.
- **Токены:** формулировка добавлена однажды, прирост длины user-сообщения небольшой.

### Рекомендации

- Оставить как есть. При желании в будущем вынести текст «Обязательно:…» в настройку (например, «Структура статьи (обязательно)») с дефолтом в коде.

## 2. humanize.ts — Промпт 3 как system-сообщение

### Реализация

- `systemContent = prompt3?.trim() || 'Перепиши текст в стиле бренда...'` — при пустом или только пробельном Промпте 3 используется встроенный дефолт.
- В API передаётся два сообщения: `system` с этой инструкцией и `user` с черновиком, ДНК и комментарием редактора.
- Из user убрана фраза «Перепиши в стиле бренда, сохрани смысл...» — она не дублируется в user.

### Проверки

- **prompt3 пустой:** `prompt3?.trim()` даёт `''`, срабатывает `||` и подставляется дефолт — ок.
- **prompt3 только пробелы:** `trim()` даёт `''`, снова дефолт — ок.
- **Промпт 3 очень длинный:** передаётся целиком в system; лимиты API на размер system-сообщения зависят от провайдера, типичные объёмы ДНК/стиля укладываются.
- **Обратная совместимость:** при пустом Промпте 3 поведение по смыслу совпадает с прежним (та же инструкция в system вместо user).

### Рекомендации

- Нет. Реализация корректна.

## 3. generation.ts — модель в статистике

### Реализация

```ts
const statsModel =
  usageDraft.model || usageHumanize.model || usageGround.model || '—';
// ...
model: statsModel,
```

В `appendStatistics` передаётся `statsModel` вместо хардкода `'DeepSeek+Sonar'`. Приоритет: черновик → очеловечивание → граундинг; если ни в одном usage нет `model`, в лист пишется `'—'`.

### Проверки

- **Откуда берётся model в usage:** в draft.ts, humanize.ts и grounding.ts в объект usage записывается переменная `model` (идентификатор модели, переданный в `create`). Тип `TokenUsage` допускает `model?: string`, но в текущем коде все три пайплайна всегда задают `usage.model`. Таким образом, в нормальном прогоне `statsModel` будет непустой строкой (например, `deepseek/deepseek-chat`).
- **Все три model отсутствуют:** теоретически возможно при изменении одного из AI-модулей; тогда в ячейку попадёт `'—'`, что приемлемо для отображения.
- **StatRow.model:** тип `string`; `statsModel` всегда строка — типы совместимы.

### Рекомендации

- Нет. Логика и типы согласованы.

## 6. Риски и ограничения

- **Поведение модели:** добавление формулировки «Обязательно: начни с заголовка…» в draft повышает вероятность соблюдения, но не гарантирует его на 100%; при необходимости можно усилить формулировкой или повтором в Промпте 2.
- **Колонка «Модель» в Статистике:** отображается одна модель (приоритет — черновик). Если граундинг и текст от разных моделей, в одной строке статистики видна только текстовая модель; разбивка по шагам потребовала бы изменения схемы листа.

Критичных ошибок не выявлено. Изменения готовы к использованию в проде.
