## 1. Fix 1 — UTM и плейсхолдер `{keyword}`

### Реализация

| Файл | Изменение |
|------|-----------|
| `src/utils/utm.ts` | Параметр `keyword: string = ''`, подстановка `{keyword}` через `encodeURIComponent(keyword.trim())` |
| `src/pipeline/generation.ts` | Вызов `buildUtmUrl(headline, settings, task.keyword ?? '')` |
| `src/utils/utm.test.ts` | Тест на `{keyword}` + в `makeSettings` добавлены `generationTime`, `publishIntervalMin` |

### Проверки

- **Единственный вызов** `buildUtmUrl` — в `generation.ts`; передаётся `task.keyword`, для задачи с заголовком он всегда задан (строка из «Ключевое слово»).
- **Обратная совместимость:** третий аргумент опциональный (`= ''`), старые вызовы с двумя аргументами остаются валидными (в коде таких нет, но API не ломается).
- **Спецсимволы:** `encodeURIComponent` корректно кодирует пробелы и символы в `utm_content`.
- **Пустой keyword:** даёт пустую строку в UTM, не ломает URL.

### Рекомендации

- Нет. Реализация соответствует ТЗ.

## 2. Fix 2 — Постфильтрация ключевых запросов по лимиту частотности

### Реализация

| Файл | Изменение |
|------|-----------|
| `src/pipeline/semantics.ts` | После `generateHeadlines()` строится `validKwSet` из `kwStrings` (Wordstat уже отфильтрован по частотности). Для каждого `item` оставляются только `item.keywords`, входящие в `validKwSet`; если после фильтра список пустой — подставляются `kwStrings.slice(0, 10)`. Дальше везде используются `filteredItems`. |

### Проверки

- **Типы:** `filteredItems` — массив объектов `{ headline, keywords }`, совпадает с `HeadlineItem`; `insertTaskRows(task, filteredItems.slice(1), costPerRow)` корректен.
- **Сравнение:** `validKwSet.has(k.trim().toLowerCase())` — нормализация по регистру и пробелам, дубликаты из AI не раздувают список.
- **Пустой Wordstat:** если `kwStrings.length === 0`, то `validKwSet` пуст, все AI-КЗ отфильтровываются, в fallback попадает `kwStrings.slice(0, 10)` = `[]` — в таблицу пишутся пустые КЗ; логика согласована с отсутствием валидных запросов.

### Рекомендации

- Имеет смысл при желании логировать факт срабатывания fallback (когда `filtered.length === 0` и подставляются до 10 из `kwStrings`), чтобы в поддержке видеть такие кейсы. Не обязательно для корректности.

## 3. Fix 3 — Расписание: время генерации и интервал публикации

### Реализация

| Файл | Изменение |
|------|-----------|
| `src/types/index.ts` | В `Settings`: `generationTime: string`, `publishIntervalMin: number`. |
| `src/sheets/settings.ts` | Чтение «Время генерации» (default `05:00`), «Интервал публикации (мин)» (default `60`); `publishIntervalMin` через `Math.max(1, parseInt(..., 10) || 60)`. |
| `src/scheduler/scheduler.ts` | `isAfterTime(timeStr)` по аналогии с `isAfterSummaryTime`. Для «Текст готов, ждём картинку» генерация картинки выполняется только при `canRunImageGeneration = isAfterTime(settings.generationTime)`. Текст генерируется сразу при статусе «Согласован заголовок» (без time-gate). Для «Одобрено на публикацию» за одну итерацию публикуется не более одной статьи; следующая — только если прошло `publishIntervalMin` минут (`lastPublishedAt`). При смене дня `lastPublishedAt` сбрасывается. |

### Проверки

- **Генерация:** Ограничение по времени только для генерации картинки (статус «Текст готов, ждём картинку»). Генерация текста («Согласован заголовок», «На доработку») и «Перегенерировать картинку» выполняются в любой момент — по смыслу так и задумано (реакция на правки редактора).
- **Публикация:** За итерацию цикла публикуется 0 или 1 статья; при `lastPublishedAt === 0` первая публикация разрешена сразу (интервал не ждём). После каждой публикации `lastPublishedAt = Date.now()`.
- **Смена дня:** При `today !== lastDateKey` обнуляются `publishedToday`, `lastPublishedAt`, чтобы лимит и интервал считались заново для нового дня.
- **Парсинг времени:** `timeStr.split(':').map((x) => parseInt(x, 10) || 0)` — при формате `"HH:MM"` или `"H:MM"` работает. Некорректные значения (например `"25:00"`) не валидируются — дают неверное время. При желании можно добавить проверку `h in 0..23, m in 0..59`.
- **Пустое «Время генерации»:** если в ячейке пусто, `get('Время генерации', '05:00')` в таблице даёт `''`. Тогда `isAfterTime('')` → `targetM = 0`, то есть «уже прошло 00:00» — генерация разрешена всегда. Рекомендация: при чтении настроек приводить пустую строку к `'05:00'`, чтобы поведение совпадало с дефолтом.
- **Интервал:** `Math.max(1, ...)` не даёт использовать 0 или отрицательные значения.

### Часовой пояс

- `isAfterTime` и `isAfterSummaryTime` используют `new Date()` — локальное время сервера. Если приложение крутится в UTC, «05:00» будет 05:00 UTC. Для московского времени нужен либо сервер в МСК, либо явная работа с таймзоной (например, настройка «Часовой пояс» и расчёт времени по нему). Сейчас поведение такое же, как у «Время сводки».

## 4. Общие замечания

### Тесты

- `npm run test` — проходят (в т.ч. `utm.test.ts` с новым тестом на `{keyword}`).
- `tsc --noEmit` — без ошибок.
- Юнит-тестов для `semantics.ts` (постфильтрация) и `scheduler.ts` (время/интервал) нет — при желании можно добавить для регрессий.

### Документация для заказчика

- В листе «Настройки» нужно завести строки (если ещё нет):
  - **Время генерации** — например `05:00`; после этого времени запускается генерация **картинки** по задачам в статусе «Текст готов, ждём картинку». Текст генерируется сразу при «Согласован заголовок».
  - **Интервал публикации (мин)** — например `60`; одобренные статьи публикуются не чаще чем раз в N минут.
- Если строки отсутствуют, используются значения по умолчанию: `05:00` и `60`.
